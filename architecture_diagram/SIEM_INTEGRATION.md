# SIEM and Log Integration Details

## Security Onion Components

### 1. Suricata (IDS)
- **Type**: Network-based Intrusion Detection System
- **Purpose**: Detect malicious network traffic using signatures
- **Deployment**: Passive sensors on vmbr1 (attacker) and vmbr2 (victim)
- **Rule sets**: ET Open, Snort community rules, custom lab rules
- **Output**: Eve JSON logs → Elasticsearch

### 2. Zeek (NSM)
- **Type**: Network Security Monitoring
- **Purpose**: Generate rich network metadata for investigation
- **Deployment**: Passive sensors on vmbr1 and vmbr2
- **Log types**: `conn.log`, `dns.log`, `ssl.log`, `http.log`, `files.log`, `x509.log`, and more
- **Output**: Zeek logs → Elasticsearch

### 3. Wazuh (SIEM / HIDS)
- **Type**: Security Information and Event Management / Host-based IDS
- **Purpose**: Centralized log aggregation, rule-based alerting, compliance tracking
- **Deployment**: Central server running inside Security Onion
- **Agents**: Installed on all Windows victims and Kioptrix (Linux)
- **Features**:
  - Log aggregation from multiple endpoint sources
  - Rule-based alert generation
  - Vulnerability management scanning
  - Compliance framework mapping (PCI-DSS, HIPAA, GDPR)
  - Real-time dashboard visualization

### 4. Elasticsearch + Kibana
- **Type**: Search and analytics engine + visualization layer
- **Purpose**: Store, index, and visualize all log data
- **Features**:
  - Full-text search across all log sources
  - Time-series analysis and trend detection
  - Custom dashboard creation
  - Alerting on custom Elasticsearch queries
  - Cross-log source correlation

## Log Sources Configuration

### Windows Event Forwarding (WEF)
```
Windows 10 / Windows 11 / Windows Server 2022
  → WinLogBeat (Elastic Beats agent)
  → Security Onion WEC (Windows Event Collector)
  → Wazuh Manager
  → Elasticsearch
```

**Key Sysmon Event IDs collected:**

| Event ID | Description |
|----------|-------------|
| 1 | Process Creation |
| 3 | Network Connection |
| 7 | Image Loaded (DLL) |
| 8 | CreateRemoteThread |
| 11 | FileCreate |
| 12/13 | Registry Object / Value Set |
| 22 | DNS Query |

### Linux Syslog (Kioptrix)
```
Kioptrix (Linux)
  → Rsyslog
  → Security Onion Syslog Receiver (UDP 514)
  → Wazuh Manager
  → Elasticsearch
```

**Log categories collected:**
- Auth logs — SSH login attempts, sudo usage, PAM events
- System logs — service start/stop, kernel messages
- Application logs — Apache access/error logs, SSH daemon logs

### Network Sensors
```
Suricata / Zeek on vmbr1 and vmbr2
  → Eve JSON / Zeek log files
  → Elasticsearch (via Filebeat or Logstash)
  → Kibana Dashboard
```

## Detection Rules Strategy

### Rule Categories

1. **Network-based Detection**
   - IDS signatures from Suricata (ET Open, Snort rules)
   - Protocol anomalies detected by Zeek analyzers
   - Traffic pattern analysis (volume, timing, direction)

2. **Host-based Detection**
   - Process behavior monitoring via Sysmon Event ID 1
   - File system activity via Sysmon Event ID 11
   - Registry modification detection via Sysmon Event ID 13

3. **Correlation-based Detection**
   - Multi-event patterns across sources
   - Timeline analysis correlating host and network events
   - Cross-source correlation (e.g., Sysmon process + Zeek connection)

## Alert Handling Workflow

```
[Alert Generated by Detection Rule]
    ↓
[Severity Assessment]
    ├─ Critical (8–10): Immediate escalation
    ├─ High (6–8):     Urgent investigation
    ├─ Medium (4–6):   Schedule investigation
    └─ Low (1–3):      Log for trending
    ↓
[Alert Enrichment]
    - Add context from threat intelligence
    - Correlate with historical data
    - Check for similar past incidents
    ↓
[Investigation Phase]
    - Analyst reviews alert details
    - Collects relevant logs from all sources
    - Builds chronological attack timeline
    ↓
[Response Phase]
    - Document findings in investigation case
    - Create incident case in Security Onion
    - Recommend detection and response improvements
    ↓
[Closure]
    - Final report generated
    - Lessons learned documented
    - Detection rules updated if needed
```

## Custom Detection Rule Examples

### Example 1: Suspicious Process Execution
```
Rule: Detect credential dumping tools
Source: Sysmon Event ID 1
Conditions:
  - Image name matches: mimikatz.exe, sekurlsa.exe, procdump.exe
  - OR: CommandLine contains "mimikatz", "sekurlsa", "lsadump"
Alert: "Credential Dumping Tool Detected"
Severity: Critical
```

### Example 2: Unusual Data Exfiltration
```
Rule: Detect large outbound data transfer
Sources: Zeek conn.log, Suricata
Conditions:
  - Destination IP not in internal whitelist
  - bytes_out > 1 GB within 1 hour
  - Source is in victim network (10.20.20.0/24)
Alert: "Possible Data Exfiltration"
Severity: High
```

### Example 3: Lateral Movement via SMB
```
Rule: Detect SMB exploitation attempt
Sources: Suricata, Zeek smb_mapping.log
Conditions:
  - SMB traffic detected between victim hosts
  - AND: Exploit signature match (e.g., EternalBlue)
  - AND: Source is known attacker host
Alert: "SMB Exploitation Attempt"
Severity: Critical
```
